# 审查目录 (docs/review/)

本目录用于存放Phase 1及后续阶段的交叉审查记录。

---

## 目录用途

1. **记录审查过程**: 保存各角色之间的代码审查记录
2. **跟踪问题状态**: 记录发现的问题及其处理状态
3. **知识沉淀**: 积累审查经验和最佳实践

---

## 文件命名规范

### 审查记录文件

格式: `YYYYMMDD_<reviewer>_<reviewee>_<type>.md`

示例:
- `20260216_inspector_architect_framework.md` - 质量审查员审查架构师的框架代码
- `20260217_inspector_unit_core.md` - 质量审查员审查单元测试工程师的核心模块测试

### 问题跟踪文件

格式: `issues_<category>_<status>.md`

示例:
- `issues_framework_open.md` - 框架相关的待处理问题
- `issues_tests_resolved.md` - 测试相关的已解决问题

## 审查指南

### 审查重点

1. **代码质量**
   - 是否符合项目编码规范
   - 是否有适当的错误处理
   - 是否有清晰的命名

2. **测试质量**
   - 测试覆盖率是否达标
   - 测试用例是否覆盖边界情况
   - 测试是否独立且可重复

3. **文档质量**
   - 文档是否完整
   - 说明是否清晰
   - 示例是否正确

### 审查态度

- **建设性**: 提出改进建议而非批评
- **具体性**: 指出具体位置和问题
- **尊重性**: 尊重开发者的努力
